# 1. Authentication
================================================================================

## Login flow
1. User opens /login (public page).
2. Client submits username + password via react-hook-form (loginSchema validation).
3. Client calls signIn("credentials", { username, password, redirect: false }) from next-auth/react.
4. Request hits NextAuth catch-all route: /api/auth/[...nextauth] (handlers from src/lib/auth.ts).
5. Credentials provider authorize():
   - Validates input with Zod (username min 3, no spaces; password min 6).
   - Loads User by username (lowercased).
   - Rejects if user missing, status DISABLED, or bcrypt password mismatch (returns null).
   - On success: records UserActivity LOGIN (best-effort), returns { id, username, role }.
6. JWT callback stores id, role, username on token; session callback exposes them on session.user.
7. Client shows toast and navigates to /dashboard (or user may have been redirected via callbackUrl from middleware).
8. Home route (/): server-side auth() → redirect to /dashboard or /login.

## Authentication library used
- next-auth v5.0.0-beta.31 (Auth.js) — src/lib/auth.ts
- @auth/core via NextAuth Credentials provider
- Client: next-auth/react (signIn, signOut, useSession)
- Route export: src/app/api/auth/[...nextauth]/route.ts → { GET, POST } handlers

## Session strategy
- strategy: "jwt" (stateless; no Session table in database)
- Session read on server: auth() from @/lib/auth
- Session read on client: useSession() from next-auth/react

## JWT/session configuration
- secret: process.env.AUTH_SECRET (required for signing/verifying JWT)
- Custom sign-in page: pages.signIn = "/login"
- JWT payload (on sign-in): id, role, username
- Session user fields: id, role, username, name (set to username)
- Type extensions: src/types/next-auth.d.ts
- No AUTH_URL / NEXTAUTH_URL referenced in codebase (platform may set implicitly)
- No server-side session revocation list; disabled users blocked at login only

## Password hashing method
- Library: bcryptjs ^3.0.3
- Algorithm: bcrypt with salt rounds = 10
- Applied on: user create (POST /api/admin/users), user update with new password (PUT /api/admin/users/[id]), admin password reset (PATCH /api/admin/users/[id]), prisma seed
- Storage: User.password column (never returned in API JSON)

## Password reset flow
- No self-service "forgot password" or email reset flow exists.
- Admin-initiated reset only:
  - PATCH /api/admin/users/[id] — body validated by resetPasswordSchema; bcrypt hash; UserActivity PASSWORD_RESET logged.
- Password change on user create/edit:
  - POST /api/admin/users — password required on create.
  - PUT /api/admin/users/[id] — optional password in updateUserSchema (min 6 if provided).
- Guards: admin cannot disable own account; cannot delete self or last active admin.

# 2. Authorization
================================================================================

## Roles available
- ADMIN (Prisma enum Role.ADMIN)
- RECEPTIONIST (Prisma enum Role.RECEPTIONIST, default on User create)
- User status: ACTIVE | DISABLED (UserStatus); disabled users fail login

## Middleware used
- File: src/middleware.ts
- Wrapper: export default auth((req) => { ... }) — NextAuth middleware integration
- Matcher: all paths EXCEPT api, _next/static, _next/image, favicon.ico, backgrounds, and static files with extensions
- API routes are NOT protected by this middleware (authorization is per-handler)

## Protected routes (pages — login required)
Prefixes (exact or subpath):
  /dashboard
  /customers
  /invoices
  /stock
  /reports
  /admin
  /settings
  /profile

Unauthenticated access → redirect to /login?callbackUrl=<pathname>

## Admin-only routes (pages — receptionist redirected)
Prefixes:
  /reports (includes /reports/revenue)
  /admin
  /settings

Receptionist + admin-only path → redirect to /dashboard

## Public routes (pages)
  /
  /login
  Static assets under /public (e.g. /branding/*) — served without app auth

## Additional server layout guard
- src/app/(dashboard)/layout.tsx: auth() → redirect /login if no session (RSC pages)

## Protected API endpoints
See Section 4 for full list. Summary:
- requireAuth(): any authenticated ACTIVE session (401 if missing)
- requireAdmin(): role must be ADMIN (403 if receptionist)

# 6. Invoice System
================================================================================

## Invoice creation flow
1. UI: /invoices/new — invoice wizard (Zustand store + InvoiceWizard component).
2. Prefetch: GET /api/settings/billing-defaults (GST defaults).
3. Optional: GET /api/customers/[id], GET /api/catalog/subscriptions|products.
4. Customer step may POST /api/customers or search GET /api/customers?q=.
5. Submit: POST /api/invoices with invoiceSchema-validated JSON.
6. Server (requireAuth):
   - calculateInvoiceTotals from items + GST settings.
   - calculatePaymentAmounts; validate partial payment rules.
   - Transaction:
     * Resolve/link/create Customer (by customerId or mobile).
     * InvoiceSequence upsert for year → formatInvoiceNumber (MRA-YYYY-#####).
     * Create Invoice + InvoiceItem rows; set createdById = session user.
   - Post-commit: CustomerActivity (if linked), UserActivity INVOICE_CREATED.
7. Response 201 → client navigates to invoice detail.

## PDF generation flow
1. User opens link: GET /api/invoices/[id]/pdf (requireAuth).
2. Load active invoice + items; load Settings id="default" (or defaults).
3. resolvePdfBrandingSettings() — resolve logo/footer/signature URLs using request origin.
4. generateInvoicePdfBuffer() — @react-pdf/renderer renderToBuffer(InvoicePDFDocument).
5. Return application/pdf inline (filename = invoiceNumber.pdf).
6. runtime = "nodejs"; dynamic = "force-dynamic".

## PDF storage method
- Invoice PDFs are NOT persisted to disk or database.
- Generated on every request (ephemeral buffer stream).
- Branding source files live in public/branding/ (static).
- Invoice business data stored in PostgreSQL (Invoice, InvoiceItem).

## Related operations
- Soft delete: DELETE /api/invoices/[id] sets deletedAt (excluded from active queries).
- Duplicate: POST /api/invoices/[id]/duplicate creates new invoice with new number/date.

# 8. Security Controls Currently Implemented
================================================================================

## Authentication controls
- Username/password credentials via NextAuth
- Zod validation on login input
- bcrypt password hashing (10 rounds)
- Disabled user accounts blocked at login (UserStatus.DISABLED)
- Generic client error on failed login (no user enumeration detail)
- JWT signed with AUTH_SECRET
- Fail-closed authorize (null → invalid session)

## Authorization controls
- requireAuth() on all business APIs (401)
- requireAdmin() on admin/reports/settings APIs (403)
- Page middleware: login required for dashboard paths
- Page middleware: receptionist blocked from /reports, /admin, /settings
- RSC dashboard layout auth() redirect
- Admin-only customer DELETE
- Profile PATCH scoped to session user id only
- User delete guards (not self, not last admin, FK on invoices)
- Admin cannot disable own account

## Validation controls
- Zod schemas: login, customer, invoice, settings, user CRUD, stock entry, catalog, UI preferences (src/lib/validations/, src/lib/ui-preferences.ts)
- Prisma parameterized queries (ORM)
- Invoice payment amount vs grand total checks
- Stock PDF type and size validation
- JSON body parse error handling on APIs

## Logging controls
- UserActivity, CustomerActivity, StockActivity (see Section 7)
- Prisma logging: error (+ warn in development)
- console.error on API failures; console.warn on skipped activity logs
- Stock bill access logs IP (x-forwarded-for / x-real-ip) and user-agent

## File upload controls
- PDF-only for stock bills
- 10 MB max size
- Filename sanitization
- Path traversal prevention on read
- Uploads directory gitignored
- Bills served only through authenticated API (not public URL)

## Session controls
- JWT strategy (no DB session table)
- Session cookie managed by Auth.js / NextAuth
- signOut() on client
- Middleware redirects unauthenticated users from protected pages
- No explicit idle timeout or token rotation in application code

## Not implemented
- MFA, OAuth/SSO, CAPTCHA, rate limiting, API gateway middleware auth, CSRF tokens on custom APIs (NextAuth handles its own), field-level encryption, WAF, automated security scanning in repo

# 10. Deployment Architecture
================================================================================

## Hosting provider
- Not specified in repository (no Dockerfile, vercel.json, or IaC).
- Application is a standard Next.js 16 Node app: npm run build && npm run start.
- Typical deployment: Node VM, container platform, or serverless Node host (PDF route requires Node runtime).

## Database provider
- PostgreSQL (Prisma datasource).
- README documents Neon-style setup (DATABASE_URL pooled + DIRECT_URL for migrations).
- Migrations: prisma/migrations/* deployed via npm run db:deploy.

## Storage provider
- Invoice PDFs: none (generated in memory).
- Stock bills: local filesystem uploads/stock-bills/ on application server (not cloud object storage in code).
- Static branding: public/ directory served by Next.js.
- Implication: multi-instance deployments need shared volume or external object storage (not implemented).

## HTTPS configuration
- Not configured in application code.
- Expected: TLS termination at reverse proxy / hosting platform (production).
- Development: commonly http://localhost:3000.

## Backup strategy
- Not defined in application codebase.
- Operational expectation: PostgreSQL provider backups/snapshots (e.g. Neon automated backups).
- uploads/stock-bills/ backups are operator responsibility (directory gitignored).

## Build & run
- Node.js 18+ (README)
- Build: prisma generate && next build
- Start: next start
- Seed: npx prisma db seed (default users — change passwords in production)
